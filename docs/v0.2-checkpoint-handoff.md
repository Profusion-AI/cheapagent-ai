### The publish saga (for institutional memory)

Three stacked failures, all fixed:

1. The repo Actions secret was created as `NMP_TOKEN` (transposed). GitHub
   cannot rename secrets; a workflow fallback bridged it, later removed.
2. Re-running a tag-triggered workflow uses the workflow file from the tag's
   commit — workflow fixes require a **new tag** (additive `v0.2.0-republish*`
   tags were used; never rewrite pushed tags).
3. `npm publish --provenance` returns E422 unless `package.json
   repository.url` matches the publishing repo. The field was missing.

Trusted publishing is now configured on npmjs.com (Profusion-AI/doc2toon,
`publish.yml`) and proven by the tokenless 0.2.1 publish. The `NMP_TOKEN`
secret and the underlying npm token are unreferenced and safe to
delete/revoke. Optional tidy-up: delete the `v0.2.0-republish` and
`v0.2.0-republish2` tags on GitHub.

### Verified end-to-end in production (real account)

Signup → confirmation-link login → quota GET → 2,815-char debit POST with
to-the-character accounting ("12,186 of 15,000 chars left today") → honest
"Keep Markdown." verdict. The graceful-degradation path (usage service down →
anonymous limit) was exercised under a real outage. Consent flows, mobile
layout, and the privacy page verified in a live browser.

### Bugs found and fixed along the way

- `consistency: "strong"` throws `BlobsConsistencyError` in lambda-compat
  functions (no `uncachedEdgeURL`) — took the quota service down with 502s.
  Eventual reads + conditional writes are the correct pattern there.
- Component `display` rules silently defeat the `hidden` attribute (orphan
  sign-out icon; sign-in button visible while logged in). Fixed globally with
  `[hidden] { display: none !important; }`.
- Textarea `maxlength="1000"` would have capped signed-in users despite the
  15,000/day quota. Removed; the dynamic JS limit governs.
- After enabling Identity, `/.netlify/identity` served 200 on `*.netlify.app`
  but 404 on the custom domain until a redeploy rebound the edge config.

## Binding constraints for all future work

The public privacy copy makes hard promises. Any new feature must hold these:

- Document bodies never leave the browser; the quota service receives
  character counts only.
- **Anonymous use makes no network requests** with content or counts.
- Per-user record is minimal (id, email, plan, today's counters, timestamps);
  previous days' counters are discarded per request; **no conversion history**.
- No advertising or analytics cookies; consent banner says "No tracking here";
  localStorage holds at most: Identity session, theme (consent-gated), consent
  choice.
- Any change to what is stored must be disclosed on the privacy page with a
  dated changelog entry (the page promises this).

## Task 7 — v0.2.1: aggregate analytics counters (red-teamed design)

First-party, server-side daily counters in the existing Blobs store. No new
vendors, no client changes, no per-user data added. ~6–8 hours.

- **Schema**: `metrics/daily/YYYY-MM-DD` in the `cheapagent-usage` store,
  one ~300-byte JSON of integers: `quota_checks`, `unique_users` (signed-in
  DAU, derived from the winning CAS write whose pre-image lacked
  `daily[today]`), `unique_converters` (first allowed POST of the day),
  `new_profiles`, `debit_attempts`, `debits_allowed`, `debits_blocked`,
  `debit_failures_503`, `chars_debited`.
- **Mechanics**: a `recordDailyMetrics()` helper mirroring the existing CAS
  loop, called once per request at terminal outcomes only (never before a
  `continue`). Retry **only on `modified:false`** — a thrown error aborts
  (undercount, never double-apply). Time-box with `Promise.race` (~1s, max 2
  attempts) so analytics can never fail or slow a user request. One PII-free
  `console.warn` on failure for observability.
- **Production-gate**: skip or namespace writes when
  `process.env.CONTEXT !== "production"` so branch-deploy tests cannot
  pollute the series.
- **No Identity event functions** (signup/login hooks sit on the signup
  critical path of a not-recommended product). Signups/day come from the
  Identity dashboard user list.
- **Privacy amendments (ship together, not after)**: amend privacy.html's
  "that is the whole list" and "previous days are discarded" sentences to
  disclose site-wide daily totals; note that on low-traffic days a daily total
  may correspond to a single account, and offer to zero low-n days on deletion
  requests; bump "Last updated"/version line; dated CHANGELOG entry. Add a
  retention cap (e.g. delete metrics blobs older than ~400 days).
- **Read path**: Netlify dashboard Blobs viewer or
  `netlify blobs:get cheapagent-usage metrics/daily/<date>`. No admin
  endpoint until friction is demonstrated.
- **Cannot answer** (by design): anonymous usage, traffic sources, funnels,
  retention cohorts. Netlify Analytics ($9/mo, server-side, cookieless) is
  the only promise-compatible add-on for visitor counts — optional, Kyle's
  call; disclose under Third parties if adopted.

## Task 8 — v0.2.2: first-party sign-in form via `@netlify/identity`

Replace the `netlify-identity-widget` modal. Estimate **2–4 days**.

- **Why**: the widget renders in a same-origin iframe that automation/agents
  cannot reach by reference (proven twice in QC); it is a stock white modal
  that ignores the brand theme; and Netlify recommends `@netlify/identity`
  for new Identity work (widget and gotrue-js are "not recommended for new
  projects").
- **Scope**: signup, login, logout; `#confirmation_token` / `#recovery_token`
  URL-hash handling (the widget did this for free); password recovery; auth
  state sync with `src/main.js`; quota Bearer-token wiring. **Verify the
  lambda-compat usage function's `context.clientContext.user` still validates
  tokens issued via `@netlify/identity`** — if not, plan the token bridge
  (or accelerate migrating usage.mjs to a modern function with explicit JWT
  verification).
- **After**: update the llms.txt iframe caveat; re-run the agent-operability
  check (sign-up tab reachable by `find`/refs).

## v0.3 checkpoint scope (from doc2toon ROADMAP)

Multi-file uploads; target-aware outputs for agent instruction files; compact
Markdown rewrite output where TOON is not the right target; split-skill
recommendations; DOCX and text-based PDF support; **paid hosted convenience
tier** (CLI stays open source). README adds before/after reports.

Standing architecture decisions for v0.3:

- **Database**: graduate to Supabase Postgres (DB only — Identity keeps auth)
  when the first of these hits: (a) billing commits (build entitlements on
  Postgres from day one; the billing decision is the database decision);
  (b) a product question requires per-user history (also requires a privacy
  rewrite, which is the real cost); (c) sustained CAS contention on Blobs.
  Free-tier traps: 7-day inactivity auto-pause, $0 → $25/mo cliff.
- **UI**: the public page stays vanilla static HTML (better for humans *and*
  agents). shadcn/React enters only as a separate authenticated island
  (e.g. `/app`) for account/billing/settings surfaces, themed with the
  existing `--ca-*` tokens.
